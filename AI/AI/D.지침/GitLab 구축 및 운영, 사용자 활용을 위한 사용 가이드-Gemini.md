물론입니다. 제공해주신 상세한 규칙과 예시 문서를 분석하여, GitLab 가이드 제작을 위해 AI에게 가장 효과적으로 질문할 수 있는 **단계별 프롬프트 전략**을 설계해 드리겠습니다.

단순히 "가이드라인 만들어줘"라고 요청하는 대신, AI의 사고를 유도하고 결과물의 품질을 극대화하는 아래의 질문들을 순서대로 사용해 보세요.

-----

### 🚀 AI를 위한 단계별 프롬프트 전략: GitLab 전문가로 만들기

이 전략은 AI가 먼저 전체적인 구조와 전략을 설계하게 한 후, 각 부분을 구체화하고, 마지막으로 사용자 친화적으로 다듬는 방식입니다. 각 단계의 프롬프트는 AI가 **최고 수준의 DevOps 아키텍트**라는 역할을 맡도록 유도합니다.

-----

### **1️⃣ 단계: 전략 수립 및 목차 생성 (거시적 접근)**

가장 먼저 할 일은 AI가 문제의 복잡성을 이해하고 전체적인 청사진을 그리게 하는 것입니다. 이는 최종 결과물이 체계적이고 깊이 있게 만들어지는 기반이 됩니다.

**🤖 이 프롬프트를 AI에게 입력하세요:**

```
너는 최고 수준의 DevOps 아키텍트이자 기술 문서 전문가야. 'GitLab 종합 가이드라인' 제작을 위한 전체 전략과 목차(TOC)를 설계해 줘.

아래 원칙에 따라 작업을 수행해 줘:
1.  **문제 재정의(Problem Redefinition):** "GitLab 가이드라인 제작"이라는 막연한 문제를 "역할 기반(Role-Based) + 단계별(Stage-Based) + 통합 시나리오(Integrated Scenario)를 갖춘 지식 시스템 구축"으로 재정의하고 시작해 줘.
2.  **복잡계 해결(Complexity Resolution):** GitLab의 복잡성을 '인프라 구축', '플랫폼 운영', '팀 활용'의 3가지 핵심 레이어로 분해해서 구조를 설계해 줘.
3.  **결과물 형태:** 최종 가이드라인의 전체 구조를 한눈에 파악할 수 있는 상세한 목차(TOC) 형태로 제시해 줘. 각 파트와 챕터가 어떤 내용을 다룰지 간략한 설명도 포함해 줘.
```

**💡 이 질문의 효과:**

  * AI에게 명확한 역할과 높은 수준의 결과물을 기대한다는 점을 전달합니다.
  * 제공된 문서의 `문제 재정의`, `복잡계 해결` 같은 사고 모델을 직접적으로 활용하여 AI가 깊이 있게 생각하도록 만듭니다.
  * 가장 중요한 산출물인 \*\*목차(TOC)\*\*를 먼저 확보하여 이후의 작업이 체계적으로 진행되도록 합니다.

-----

### **2️⃣ 단계: 영역별 상세 내용 생성 (미시적 접근)**

전체 목차가 완성되면, 각 파트별로 살을 붙여나가는 단계입니다. 이때, 단순히 내용만 채우는 것이 아니라, 제공된 문서의 **디자인 패턴**을 적용하도록 구체적으로 지시하는 것이 핵심입니다.

**🤖 'Part 1: 구축' 가이드를 위해 이 프롬프트를 입력하세요:**

```
이전 단계에서 생성한 목차 중 'Part 1: 구축 (Infrastructure Setup)'에 대한 상세 내용을 작성해 줘.

아래의 디자인 패턴을 반드시 적용해 줘:
1.  **실행 가능성(Actionability):** 모든 코드와 명령어는 독자가 바로 복사-붙여넣기 해서 사용할 수 있도록 완성된 스크립트 형태로 제공해 줘. (예: 변수 부분을 `YOUR_DOMAIN`으로 표기)
2.  **검증 가능성(Verifiability):** 모든 절차는 '사전 준비 체크리스트 → 실행 단계 → 결과 확인 방법'의 3단계(Before-After-Verify) 구조를 따르도록 작성해 줘.
3.  **결정 트리(Decision Tree):** 다양한 설치 방법(Omnibus, Docker, Kubernetes)을 선택해야 할 때, 어떤 상황에 어떤 방법을 선택해야 하는지 기준을 명확한 의사결정 트리나 표로 제시해 줘.

특히 '1.2.A Omnibus 패키지 설치' 가이드는 초보자도 따라 할 수 있도록 스크립트 예제, 검증 단계, 초기 보안 설정, 문제 해결 FAQ를 포함해서 매우 상세하게 작성해 줘.
```

**💡 이 질문의 효과:**

  * 이전 대화의 맥락(목차)을 이어받아 일관성을 유지합니다.
  * `실행 가능성`, `검증 가능성` 같은 구체적인 **품질 기준**을 제시하여 결과물의 퀄리티를 보장합니다.
  * 가장 중요한 예시 하나를 매우 상세하게 작성하도록 요구함으로써, 다른 부분들도 해당 수준에 맞춰 생성하도록 기준점을 설정합니다.

**(동일한 방식으로 `Part 2: 운영`, `Part 3: 팀 활용`에 대해서도 순서대로 질문할 수 있습니다.)**

-----

### **3️⃣ 단계: 최종 요약 및 활용도 향상 (사용자 경험 최적화)**

모든 상세 내용이 작성되었다면, 최종적으로 사용자가 문서를 쉽게 이해하고 활용할 수 있도록 다듬는 단계입니다.

**🤖 이 프롬프트를 AI에게 입력하세요:**

```
지금까지 작성된 모든 GitLab 가이드라인 내용을 종합하여, 사용자가 최종적으로 활용할 '종합 요약본'과 '활용도 향상 자료'를 생성해 줘.

결과물은 아래 항목들을 포함해야 해:
1.  **역할별 체크리스트 (Final Checklist):** 관리자, DevOps 엔지니어, 개발자 각 역할별로 GitLab을 사용하기 전 반드시 확인해야 할 필수 체크리스트를 만들어 줘.
2.  **GUI 스타일 요약 (GUI-Style Summary):** 전체 가이드의 핵심 내용을 카드, 테이블, 접기/펴기(Accordion) 기능을 활용한 시각적인 대시보드 형태로 요약해 줘. 각 카드에는 해당 상세 가이드로 바로 갈 수 있는 링크(목차명)를 포함해 줘.
3.  **추가 권장 사항 (Additional Recommendations):** 가이드의 활용도를 높이기 위한 영상 튜토리얼 추천 링크, 간단한 실습 환경 구축용 Docker Compose 스크립트, 자주 묻는 질문(FAQ) Top 5 등 추가적인 액션 아이템을 제안해 줘.
```

**💡 이 질문의 효과:**

  * 단순한 텍스트의 나열을 넘어 **실용적인 최종 산출물**을 얻을 수 있습니다.
  * `체크리스트`, `GUI 스타일 요약` 등 시각적이고 인터랙티브한 결과물을 요구하여 정보의 가독성과 이해도를 높입니다.
  * 문서 자체로 끝나는 것이 아니라, 사용자가 다음 행동으로 이어질 수 있도록 돕는 추가 자료까지 확보할 수 있습니다.
