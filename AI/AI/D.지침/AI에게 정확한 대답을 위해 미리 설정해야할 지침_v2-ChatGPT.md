# **AI에게 정확도를 높이는 ‘사전 지침(Instruction) 설계 매뉴얼’**

## **A. 왜 지침이 중요한가**

* Overview

  * ▷ 배경

    * 모델은 “당신이 준 목표·역할·출력 형식”에 강하게 조건화됨
    * 지침이 구체적일수록 모호성↓, 일관성↑, 재작업↓
    * 다중 모델(ChatGPT·Claude·Perplexity·Gemini) 간 **공통 코어 + 소량의 모델별 차등**이 효율적
    * → 본 문서는 **바로 복붙 가능한 템플릿 + 체크리스트 + 예시**를 GUI 스타일로 제공

> **Key takeaway: 코어 지침은 ‘역할·목표·범위·제약·출력 포맷·평가 기준’ 6요소로 고정하고, 모델별로 얇게 튜닝한다.**

---

## **B. 핵심 설계 원칙 (THINK HARD / STEP-BY-STEP 내장)**

* ▷ 원칙 6

  1. **역할(Role)**: 전문가 페르소나 + 말투/톤
  2. **목표(Goal)**: 이번 대화의 산출물 정의(정확성의 기준 포함)
  3. **범위(Scope)**: 포함/제외, 입력 가정(Assumptions)
  4. **절차(Process)**: “Think hard → Step-by-step → Self-check”를 **명문화**
  5. **출력(Output)**: 구조(섹션·표·코드), 분량, 예시/근거, 파일형식
  6. **평가(Eval)**: 성공 기준, 실패 신호, 재시도 규칙(리루프)
* ▷ 금지 4

  * 모호한 목표, 암묵적 제약, 포맷 미지정, 검증 단계 누락

> **Key takeaway: ‘6요소 정의 + 4금지 회피’가 정확도 절반을 먹고 들어간다.**

---

## **C. 즉시 사용 가능한 ‘모델 공통’ 지침 템플릿**

아래 블록을 **System/Instructions**에 그대로 붙이고, 굵은 대괄호 항목만 교체하세요.

```text
[ROLE]
너는 [분야/직무] 전문가다. 초보도 이해할 수 있게 친근하지만 정확하게 설명한다.

[GOAL]
이번 대화의 목표는 [구체 산출물]을 만드는 것이다. 성공 기준:
- 정확성: [정확성 기준/근거 형태]
- 완결성: 누락 없음, 필요한 경우 가정 명시
- 재현성: 단계별 절차 + 예시 데이터/결과 포함

[SCOPE]
포함: [리스트] / 제외: [리스트]
가정: [명시적 가정 1-3개]

[PROCESS]  (THINK HARD) (STEP-BY-STEP)
1) 문제 재정의: 사용자의 요구를 한 줄로 요약하고 누락/모순 점검  
2) 세부 설계: 하위 과업으로 분해, 우선순위 지정  
3) 풀이: 단계별로 계산/추론 과정을 **숫자/수식/근거**로 제시  
4) 자체검증: 
   - 체크리스트로 정확성/일관성 점검
   - 반례/엣지케이스 1개 이상 제시
5) 요약: 실행 가능한 TODO와 다음 액션을 표로 제공

[OUTPUT]
- 문서 구조: #제목 → ##개요 → ##절차 → ##대안 → ##체크리스트 → 부록(근거)
- 표: 핵심 지표 3~5개, 단위/기준 명시
- 코드/의사코드가 있으면 **완전 실행 가능**하게 제공(입력/출력 예 포함)
- 길어질 경우: 개요→세부(접기/펼치기 느낌의 구분) 순서

[EVAL]
- 실패 신호: (a) 기준 미충족 (b) 근거 부재 (c) 포맷 불일치
- 재시도 규칙: 실패 신호 감지 시, 가정/범위 재설정 → 대안 2–3개 비교 → 수정본 제시

[STYLE]
- 한국어, GUI 스타일(카드/체크리스트/표/요약 배지)
- 과장/모호 표현 금지. 수치/날짜는 절대값 표기.
```

> **Key takeaway: 공통 템플릿에 “자체검증(EVAL) + 구조화된 OUTPUT”을 강제하면 품질이 튀어 오른다.**

## **D. 모델별 미세 튜닝 포인트 (짧고 명확하게)**

* ▷ ChatGPT (GPT 계열)

  * **System**에 C 템플릿 전체 삽입 → User 질문
  * 긴 답변 시 “개요→상세” 두 단계, 표/코드 적극 활용
  * 수학·계산은 “숫자 단위·자릿수·검산” 명시 요구
* ▷ Claude

  * “사고과정은 내부에서, 결과는 간결하게”를 추가
  * 긴 텍스트 요약/재서술·톤 컨트롤에 강점 → **톤·독자수준** 구체화
* ▷ Perplexity

  * **출처/인용 규칙**을 꼭 명시(최소 3개, 신뢰도 높은 도메인 우선)
  * “요약 후 링크 목록” 형식 지시
* ▷ Gemini

  * **제약 조건과 안전 필터 해석을 명확히**(금지·허용 사례 포함)
  * 멀티모달 요청 시 입력 형식(이미지/표)와 기대 출력 예시를 붙임

> **Key takeaway: 모델 특성에 맞춰 ‘한 줄 가드레일’을 추가하면 튜닝 대비 효과가 크다.**

---

## **E. 목적별 미니 프롬프트 6종 (복붙용)**

* 1. **정확한 사실 조회**

  * “사실만 요약해. 출처 3개, 발행일 표시, 서로 다른 도메인 우선. 모호하면 명시적으로 ‘모른다’라고 말해.”
* 2. **절차 설계/표준 운영**

  * “업무를 단계로 쪼개서 RACI 표(담당/검토/승인/참고)로 보여줘. 리스크·완화책 포함.”
* 3. **데이터 해석**

  * “통계 검정 가정 충족 여부부터 체크하고, 유의확률·효과크기·신뢰구간을 표로 내.”
* 4. **코드 생성**

  * “완전 실행 스크립트, 입출력 예 2개, 실패 케이스 1개, 복잡도/메모리 코멘트 포함.”
* 5. **의사결정 대안 비교**

  * “옵션 3개, 가중치 점수표(기준 4개), 민감도 분석 1회, 추천 1줄.”
* 6. **문서 요약**

  * “5문장 TL;DR → 핵심 숫자/날짜 → 액션 아이템 3개 순서로.”

> **Note:** 미니 프롬프트는 **C 템플릿과 함께** 쓰면 더 강력하다.

## **F. 예시: 일상생활 용 ‘맞춤 지침’ (바로 사용)**

```text
[ROLE] 너는 생활 코치 겸 리서치 어시스턴트다.
[GOAL] 내가 내린 선택의 질을 높이기 위한 ‘사실 검증 요약 + 실행 체크리스트’를 만든다.
[SCOPE] 포함: 일정/비용/장단점; 제외: 광고성 추천. 가정: 한국 거주, KST 기준.
[PROCESS] (THINK HARD)(STEP-BY-STEP)
1) 질문 재정의 → 2) 정보 수집 포인트 제시 → 3) 비교표 작성(3옵션) 
4) 리스크·대안 → 5) 최종 권고 1문장 → 6) TODO 체크리스트
[OUTPUT] 카드 요약, 표 1개(지표 4개), 날짜·단위 절대값, 링크 모음.
[EVAL] 출처 3개 이상, 상충 정보 있으면 대비표로 명시, 누락 시 재시도.
[STYLE] 간결/명확, 과장 금지.
```

## 🔁 자체검증
- 불일치/누락 발견 시 수정 사항:

## **H. 자주 실패하는 패턴 → 교정문**

* “대충 알아서 해줘” → **교정:** 목표·출력 포맷·평가 기준 3가지는 반드시 명시
* “정답만” → **교정:** 근거/계산/출처를 한 섹션으로 강제
* “긴 글 써줘” → **교정:** 길이 대신 **구조**(개요→상세→체크리스트) 지정
* “최신만” → **교정:** “발행일/수정일 표기 + 상충 시 대비표” 요구

> **Key takeaway: 목표·근거·구조를 명문화하면 실패 패턴이 사라진다.**

---

## **I. 단계별 적용 가이드 (처음 쓰는 사람용)**

1. **C 템플릿**을 System/지침란에 복붙 → 대괄호 채우기(2분)
2. 목적별 **E 미니 프롬프트** 중 해당 항목 1개 추가
3. 대화 시작 전 **G 포맷**을 “이 포맷으로 출력”이라 명시
4. 첫 응답에 **H 교정문**으로 스스로 점검 지시
5. 필요하면 **SCOPE/EVAL**만 소폭 조정하고 고정

---

## **J. 대안 설계(2–3안)**

* **안 A(표준형):** C 템플릿 + E 미니 프롬프트 1개 → 대부분의 상황에 무난
* **안 B(근거강화형):** C + “출처 3개·상충 대비표·날짜 절대값”을 강제 → 리서치/의사결정에 적합
* **안 C(개발형):** C + “완전 실행 코드 + 테스트 케이스 + 성능지표” → 코드 산출 필수 업무용

> **Key takeaway: 표준형에서 출처·코드·테스트만 스위치로 켜고 끄면 된다.**

---

## **✅ 최종 체크리스트**

1. 역할·목표·범위가 한 줄로 명확한가
2. PROCESS에 (THINK HARD)(STEP-BY-STEP)과 자체검증이 있는가
3. OUTPUT 포맷이 카드/표/체크리스트로 고정됐는가
4. EVAL에 성공/실패·재시도 규칙이 있는가
5. 모델별 한 줄 가드레일을 추가했는가(출처/톤/안전)
6. 예시·반례·엣지케이스가 최소 1개씩 있는가
7. 날짜·수치가 **절대값**으로 표기됐는가
8. 길어질 때 개요→상세 순서가 유지됐는가

---
