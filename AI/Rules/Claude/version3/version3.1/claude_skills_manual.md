## 1. Claude Skills의 정의와 핵심 개념

### 1.1 Claude Skills란?

**Claude Skills**는 Claude라는 AI에게 특정 작업을 더 효율적이고 정확하게 수행하도록 가르치는 **맞춤형 능력 패키지**입니다. 

일반적인 프롬프트와 달리, Skills는:

- **재사용 가능** - 한 번 만들면 계속 사용
- **일관성 있음** - 매번 동일한 품질의 결과 제공
- **자동 활성화** - Claude가 상황을 판단하여 자동으로 사용
- **모듈식** - 여러 Skills를 조합하여 복합 작업 수행

### 1.2 기존 프롬프트 엔지니어링과의 차이점

#### 기존 프롬프트 방식의 한계:
- "이번엔 이렇게 해줘" - 매번 새로운 지시 필요
- "지난번 그 형식으로" - 컨텍스트 매번 반복
- 결과물의 편차 - 매번 다른 품질의 결과
- 토큰 낭비 - 동일한 지침을 계속 반복

#### Claude Skills의 장점:
- **일회성이 아닌 시스템화** - 업무 매뉴얼처럼 영구적으로 사용
- **자동 적용** - 관련 작업이면 Claude가 스스로 판단하여 사용
- **효율성** - 필요한 정보만 필요할 때 로드 (토큰 절약)
- **조직 표준화** - 팀 전체가 동일한 규칙 준수

### 1.3 왜 Claude Skills가 게임 체인저인가?

Claude Skills는 단순한 프롬프트를 넘어, **AI를 여러분의 업무 방식에 맞게 커스터마이징하는 온보딩 방식**입니다. 

마치 새 직원에게:
- 회사 규칙과 프로세스 (지침)
- 이전 작업 사례 (예제)
- 자동화 도구 (스크립트)

를 제공하는 것과 같습니다. 이를 통해 Claude는:

1. **범용 AI** → **전문가급 AI로 변신**
2. **일관성 없는 결과** → **안정적이고 신뢰할 수 있는 산출물**
3. **반복적인 수동 작업** → **자동화된 워크플로우**

를 달성할 수 있습니다.

### 2.2 Skills가 작동하는 작업의 특징

Skills가 가장 효과적인 작업들은 다음과 같은 특징을 가집니다:

✅ **구조화된 작업** - 정해진 형식과 절차가 있음
✅ **반복적인 작업** - 자주 반복되어야 함
✅ **표준화 필요** - 일관된 품질이 중요함
✅ **규칙 기반** - 명확한 규칙과 지침이 존재
✅ **여러 단계** - 복잡한 프로세스를 자동화

### 2.3 Skills가 적합하지 않은 경우

❌ 매우 창의적인 작업 (예술, 새로운 아이디어 생성)
❌ 완전히 새로운 상황에 대한 즉각적 판단
❌ 인간의 감정 이입이 필수적인 작업
❌ 빈번한 예외 처리가 필요한 업무

## 3. Claude Skills 기본 구조

### 3.1 Skills 폴더 구조

모든 Claude Skill은 다음과 같은 폴더 구조를 가집니다:

```
my-skill/
├── SKILL.md                    (필수) 스킬의 정체성과 핵심 지침
├── REFERENCE.md                (선택) 상세 참고 자료
├── examples/                   (선택) 입출력 예시 폴더
│   ├── good_input.txt
│   ├── good_output.txt
│   └── bad_example.txt
├── scripts/                    (선택) 실행 가능한 코드
│   ├── analyze_data.py
│   └── generate_report.py
├── templates/                  (선택) 문서 템플릿
│   └── report_template.docx
└── resources/                  (선택) 로고, 이미지, 설정
    ├── logo.png
    └── config.json
```

### 3.2 SKILL.md 파일 상세 설명

**SKILL.md**는 스킬의 가장 중요한 파일입니다. 이 파일은 두 부분으로 구성됩니다:

#### Part 1: YAML 프론트매터 (필수)

```yaml
---
name: your-skill-name
description: Brief but comprehensive description of what this Skill does and when to use it
version: 1.0.0
---
```

**필드 설명:**

| 필드 | 설명 | 예시 |
|------|------|------|
| **name** | 스킬의 고유 식별자 (소문자, 하이픈만 사용, 최대 64자) | `pdf-processing` |
| **description** | 스킬의 목적과 사용 시점 (최대 1024자, 매우 중요!) | "PDF 파일에서 텍스트와 테이블을 추출하고, 양식을 작성하며, 문서를 병합합니다. PDF 파일로 작업할 때, 또는 사용자가 PDF, 양식, 문서 추출을 언급할 때 사용합니다." |
| **version** | 스킬의 버전 (선택사항) | `1.0.0` |

**Description 작성의 핵심:**
- **"무엇을"** 할 수 있는가? (기능)
- **"언제"** 사용되는가? (사용 시점)
- **"어떻게"** 작동하는가? (방식)

❌ 나쁜 예: "문서를 만드는 스킬입니다"
✅ 좋은 예: "회사의 주간 업무 보고서를 '진행 중', '완료', '이슈' 세 항목으로 구분하여 생성합니다. 각 항목은 글머리 기호로 정리하며, 매주 월요일 아침에 사용합니다."

#### Part 2: 마크다운 본문 (필수)

YAML 프론트매터 아래에 실제 지침을 마크다운 형식으로 작성합니다:

```markdown
# Skill Name

## Overview
스킬의 전체 목적과 기능을 2-3문장으로 설명합니다.

## Instructions
단계별 지침을 제공합니다.

1. 첫 번째 단계
2. 두 번째 단계
3. 세 번째 단계

## Examples
실제 예시를 제공합니다.

### Example 1: Basic Usage
**Input:** ...
**Output:** ...

## Edge Cases
예외 상황과 처리 방법입니다.

## References
추가 파일이나 리소스에 대한 링크입니다.
```

### 3.3 선택적 파일들

#### REFERENCE.md
- **용도:** SKILL.md에 담기에는 너무 큰 참고 자료
- **예시:** 상세 API 문서, 100개 이상의 규칙 목록, 역사적 데이터

#### examples/ 폴더
- **용도:** Claude에게 기대되는 입출력을 구체적으로 보여줌
- **파일:** good_input.txt, good_output.txt, bad_example.txt
- **효과:** 10줄의 지침보다 1개의 좋은 예시가 더 효과적

#### scripts/ 폴더
- **용도:** 실행 가능한 코드 (Python, Bash 등)
- **효과:** 텍스트 생성을 넘어 데이터 처리, 파일 조작, API 연동 가능
- **예시:** analyze_sales.py, validate_form.py, merge_pdfs.sh

#### templates/ 폴더
- **용도:** 출력 형식의 기준이 되는 템플릿 파일
- **예시:** report_template.docx, email_template.txt

## 4. Claude Skills 작동 원리

### 4.1 Claude Skills 자동 실행 과정

Claude가 사용자의 요청을 받으면 다음과 같은 과정을 거칩니다:

#### Step 1: 요청 분석
사용자의 메시지를 분석합니다.
예: "우리 회사 브랜드 가이드라인에 맞춰서 제품 설명서 만들어줘"

#### Step 2: 스킬 스캔
설치된 모든 스킬의 **설명(description)**을 확인하여 관련성을 판단합니다.
- 사용 가능한 스킬 1: "PDF 처리"
- 사용 가능한 스킬 2: "브랜드 가이드라인 적용" ← **이것!**
- 사용 가능한 스킬 3: "이메일 템플릿"

#### Step 3: 스킬 선택 및 로드
가장 관련성 높은 스킬을 선택하여 **SKILL.md의 본문**을 컨텍스트에 로드합니다.

#### Step 4: 작업 실행
로드된 지침에 따라 작업을 수행합니다.
- 필요하면 REFERENCE.md 로드
- 필요하면 예시 파일 참조
- 필요하면 스크립트 실행

#### Step 5: 결과 생성
지침에 맞게 결과물을 생성하여 사용자에게 제공합니다.

### 4.2 점진적 정보 로딩 (Progressive Disclosure)

Claude Skills의 가장 혁신적인 특징은 **필요한 정보만 필요할 때만 로드**한다는 것입니다.

#### 🔵 레벨 1: 메타데이터 (항상 로드)
- **내용:** name, description
- **토큰 비용:** Skill당 약 100 토큰
- **로드 시점:** Claude 시작 시 한 번만

Claude는 사용 가능한 모든 스킬의 메타데이터를 항상 알고 있어서, 어떤 스킬이 무엇을 하는지 파악합니다.

#### 🟠 레벨 2: 지침 (스킬 활성화 시 로드)
- **내용:** SKILL.md의 본문 내용
- **토큰 비용:** 5,000 토큰 미만
- **로드 시점:** 사용자 요청이 스킬과 관련성이 있을 때

스킬이 선택되면 Claude는 상세한 지침을 읽고 그에 따라 작업합니다.

#### 🟡 레벨 3: 리소스 (필요시만 로드)
- **내용:** REFERENCE.md, 스크립트, 템플릿, 예시
- **토큰 비용:** 사실상 무제한
- **로드 시점:** 지침에서 참조되거나 필요할 때만

이 아키텍처의 혁신성은:
- **컨텍스트 효율성:** 필요한 것만 로드하므로 토큰 절약
- **확장성:** 수십 개의 스킬을 설치해도 시작 시간 증가 없음
- **유연성:** 무제한의 참고 자료를 추가해도 기본 성능에 영향 없음

### 4.3 여러 Skills 조합

Claude는 한 번에 여러 스킬을 조합하여 사용할 수 있습니다.

**예시:**
사용자: "올해 매출 데이터를 분석해서, 우리 브랜드 가이드라인에 맞춘 아름다운 프레젠테이션으로 만들어줘"

Claude의 반응:
1. "Excel 데이터 분석" 스킬 로드 → 데이터 분석
2. "파워포인트 설계" 스킬 로드 → 슬라이드 설계
3. "브랜드 가이드라인" 스킬 로드 → 색상, 폰트, 로고 적용
4. 최종 프레젠테이션 생성

## 5. Claude Skills 단계별 제작 가이드

### 5.1 기본 원칙: ERIC 법칙

Skills를 만들 때 항상 기억할 4가지 원칙:

| 원칙 | 의미 | 실행 방법 |
|------|------|----------|
| **E**ssential | 필수적인 것만 | 불필요한 정보는 빼기 |
| **R**elevant | 관련성 있는 내용 | 스킬과 무관한 내용 제거 |
| **I**mmediate | 즉시 실행 가능 | 복잡한 절차 간소화 |
| **C**lear | 명확한 표현 | 모호한 표현 피하기 |

### 5.2 단계 1: 스킬의 목적 정의

**첫 번째로 해야 할 일은 스킬이 "무엇을" 할 것인지 명확히 정의하는 것입니다.**

```
질문들:
- 이 스킬은 어떤 작업을 자동화하는가?
- 누가 이 스킬을 사용할 것인가?
- 이 스킬이 해결하려는 문제는?
- 이 스킬의 입력과 출력은?
- 기대되는 결과물의 형식은?
```

**예시: 브랜드 가이드라인 스킬**
```
목적: 프레젠테이션과 문서가 항상 Acme Corp의 브랜드 아이덴티티를 유지하도록 함
사용자: 마케팅팀, 영업팀, 콘텐츠팀
문제: 매번 색상, 폰트, 로고가 달라서 전문성 저하
입력: 프레젠테이션이나 문서의 초안
출력: 공식 브랜드 가이드라인이 적용된 최종 문서
```

# API 문서 생성 스킬

## 개요
소스 코드의 함수 정의를 분석하여 표준 형식의 API 문서를 자동 생성합니다.

## 문서 형식

### 기본 구조
```
## Function Name
설명 문장

**파라미터:**
- param1 (type): 설명
- param2 (type): 설명

**반환값:**
- type: 반환값 설명

**예시:**
\`\`\`python
result = function_name(param1, param2)
\`\`\`

**오류 처리:**
- ValueError: 파라미터가 유효하지 않을 때
```

## scripts/parse_api.py
```python
import ast
import inspect

def extract_api_from_code(code_string):
    """
    코드 문자열에서 함수 정보 추출
    """
    tree = ast.parse(code_string)
    apis = []
    
    for node in ast.walk(tree):
        if isinstance(node, ast.FunctionDef):
            func_info = {
                'name': node.name,
                'docstring': ast.get_docstring(node),
                'args': [arg.arg for arg in node.args.args],
                'line': node.lineno
            }
            apis.append(func_info)
    
    return apis

# 사용 예시
code = """
def calculate_discount(price, discount_percent):
    '''가격에 할인율을 적용합니다.'''
    return price * (1 - discount_percent / 100)
"""

apis = extract_api_from_code(code)
for api in apis:
    print(f"함수: {api['name']}, 파라미터: {api['args']}")
```

- 관리와 업데이트가 쉬움
- 테스트와 버전 관리 용이

#### 📝 원칙 2: 설명이 전부다! (명확한 Description)
```
❌ "문서를 만듭니다"
❌ "보고서 작성"
✅ "회사의 주간 업무 현황을 '진행 중', '완료', '이슈' 세 가지로 구분하여, 글머리 기호 형식의 보고서를 작성합니다. 매주 월요일 아침 사용하는 스킬입니다."
```

Description 작성 체크리스트:
- [ ] "무엇을" 하는가? (기능)
- [ ] "언제" 사용되는가? (상황)
- [ ] "어떻게" 작동하는가? (방식)
- [ ] 사용자가 검색할 핵심 키워드 포함

#### 👀 원칙 3: 보여주는 것이 최고의 스승! (예시 제공)
"이렇게 해줘" 10줄 > "다음 예시를 참고해" 1개

Examples를 만들 때:
- [ ] Good Example: 정확히 기대하는 결과
- [ ] Bad Example: 하지 말아야 할 것
- [ ] Edge Case: 예외 상황
- [ ] Complex Example: 복잡한 상황

#### 🌱 원칙 4: 작게 시작해서 크게 키워라! (점진적 확장)
1단계: SKILL.md만 (지침)
2단계: 테스트 후 REFERENCE.md 추가
3단계: 예시 추가
4단계: 스크립트 추가

처음부터 모든 기능을 포함하려고 하지 말 것!

#### ✅원칙 5: AI를 믿되, 검증하라! (인간 감독)
중요한 작업에서는:
- "분석 결과를 보여드릴까요? 계속할까요 취소할까요?" 형태의 검증 단계 포함
- 최종 실행 전 사용자 승인 요청
- 실행 결과 자동 검증 스크립트 포함

### 7.2 Description 작성 마스터 가이드

|--------|-------------------|

| 데이터 | "Analyze sales data, calculate trends, identify anomalies, generate reports. Use when analyzing Excel files, spreadsheets, or when the user mentions sales, metrics, or dashboard." |
| 이메일 | "Draft professional emails for customer service, complaints, follow-ups. Use when the user asks about writing customer emails or responding to feedback." |
| 코드 | "Generate API documentation from code, create README files. Use when working with Python/JavaScript code or when the user mentions API docs." |

## 8. 디버깅 및 문제 해결

### 8.1 "Claude가 스킬을 사용하지 않는 경우"

**증상:** 스킬과 관련된 요청을 했는데도 활성화되지 않음

**해결 단계:**

1️⃣ **Description 확인**
```
현재: "문서를 만듭니다"
변경: "PowerPoint 프레젠테이션을 작성하거나, Word 문서를 생성하거나, PDF 형식으로 변환할 때 사용합니다."
```

2️⃣ **키워드 포함 확인**
Description에 사용자가 쓸 만한 구체적 단어 포함:
- ❌ "데이터"
- ✅ "Excel, CSV, 스프레드시트, 매출, 분석"

3️⃣ **YAML 문법 확인**
```bash
# 파일의 처음 10줄 확인
cat SKILL.md | head -n 10
```
- `---`로 시작하는가?
- `---`로 종료하는가?
- 쌍따옴표가 일치하는가?

### 8.2 "스킬이 로드되었지만 결과가 기대와 다른 경우"

**문제:** 스킬은 활성화되지만 결과가 요청과 맞지 않음

**해결 단계:**

1️⃣ **지침 명확화**
```
현재:
"명확한 구조로 보고서를 작성하세요"

변경:
"다음 구조를 따르세요:
1. 제목 (H1)
2. 요약 (100단어 이내)
3. 상세 내용 (섹션별 H2)
4. 결론 (명확한 액션 아이템 포함)"
```

2️⃣ **예시 추가**
```
examples/good_report.md:
정확히 기대하는 형식의 보고서를 파일로 제시
```

3️⃣ **체크리스트 추가**
```markdown
## 검증 체크리스트
- [ ] 제목이 명확한가?
- [ ] 모든 섹션이 포함되어 있는가?
- [ ] 데이터가 정확한가?
- [ ] 형식이 표준을 따르는가?
```

### 8.3 "여러 스킬이 경쟁하는 경우"

**문제:** 비슷한 2개의 스킬 중 잘못된 것이 활성화됨

**해결:**
```
스킬 A Description:
"분석: Sales data를 분석하여 매출 현황을 파악합니다."

스킬 B Description:
"분석: Log 파일을 분석하여 시스템 오류를 탐지합니다."

사용자: "데이터 분석해줘"
→ 애매함!

해결: Description을 명확히 구분
스킬 A: "Sales data 파일(Excel, CSV) 분석 - 매출, 트렌드, CRM 데이터"
스킬 B: "Log 파일 분석 - 오류 탐지, 성능 모니터링"
```

### 8.4 "스크립트 실행 실패"

**문제:** 스킬의 scripts/ 폴더의 코드 실행 시 오류

**확인 사항:**
```bash
# 1. 파일 권한 확인
ls -la scripts/
# 실행 권한 추가
chmod +x scripts/process.py

# 2. 의존성 확인
# SKILL.md에 필수 패키지 명시
pip install pandas numpy matplotlib

# 3. 경로 확인
# 절대 경로 대신 상대 경로 사용
# ❌ /home/user/project/data.csv
# ✅ data.csv
```

## 부록: 자주 묻는 질문 (FAQ)

### Q1: 스킬을 만들려면 코딩을 반드시 알아야 하나요?
**A:** 아닙니다! 기본 스킬은 마크다운 파일만으로 만들 수 있습니다. 코딩은 선택사항이며, 데이터 처리 같은 복잡한 작업이 필요할 때만 스크립트를 추가합니다.

### Q2: 한 번 만든 스킬을 다시 수정할 수 있나요?
**A:** 물론입니다! SKILL.md 파일을 편집한 후 다시 업로드하면 됩니다. 버전을 명시하여 변화를 추적하는 것도 좋은 방법입니다.

### Q3: 내 스킬을 다른 사람과 공유할 수 있나요?
**A:** 가능합니다. 스킬 폴더를 ZIP 파일로 압축하여 공유하거나, 팀원들이 공유 프로젝트에 접근하도록 설정할 수 있습니다. Claude Code에서는 프로젝트 스킬이 자동으로 팀과 공유됩니다.

### Q4: 스킬의 성능을 어떻게 측정하나요?
**A:** 다음 지표로 측정할 수 있습니다:
- 사용 빈도 (Claude의 "생각" 프로세스에서 확인)
- 사용자 만족도 (피드백 수집)
- 시간 절약 효과 (이전 vs 이후 비교)

### Q5: 스킬이 작동하지 않으면 어떻게 해요?
**A:** 다음 순서로 확인하세요:
1. Description이 구체적인지?
2. YAML 문법이 정확한지?
3. 파일 구조가 올바른지?
4. 테스트 프롬프트가 관련이 있는지?
